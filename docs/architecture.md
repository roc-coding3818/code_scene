# 架构设计文档

## 设计原则

### 1. 模块独立原则
- 每个业务场景独立成模块
- 模块可单独编译、运行、测试
- 模块间通过 API 通信，不直接依赖

### 2. 分层架构
```
api (接口层) → service (业务层) → repo (数据层)
```

### 3. 面向接口编程
- 各层之间通过接口定义
- 便于单元测试和替换实现

## 模块划分

### 限界上下文（Bounded Context）

| 模块 | 限界上下文 | 核心域 |
|------|-----------|--------|
| user | 用户中心 | 用户、角色、权限 |
| order | 订单中心 | 订单、订单项 |
| payment | 支付中心 | 支付、退款 |
| product | 商品中心 | 商品、分类、SKU |

### 模块间依赖

```
user ──→ order ──→ payment
  │         │
  └─────────┴──→ product
```

## 技术选型

### Web 框架
- **Gin**: 高性能、轻量级

### ORM
- **GORM**: 功能完整、使用广泛

### 缓存
- **Redis**: 缓存、会话、分布式锁

### 消息队列
- **RabbitMQ**: 可靠消息传递
- **Kafka**: 高吞吐场景

### 搜索
- **ElasticSearch**: 全文搜索

## 扩展性设计

### 水平扩展
- 无状态服务，可多实例部署
- Nginx 负载均衡

### 垂直扩展
- 模块独立，可按需部署
- 资源配置可动态调整
